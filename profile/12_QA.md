# Q&A（常见问题速查）

> 用途：把对话里高频疑问（故事线、产物、`null`、评测口径）整理成可直接回答审稿人/组会的问题，并指向对应文档位置。

## Q1：那句 “Semantic Knowledge Distillation” 到底是什么意思？

**A：可以拆成三层：**

1. **one-hot 的问题**：传统监督把每个疾病/属性当作彼此独立的维度，只强调“对/错”，不鼓励模型学习概念之间的相似性与共现结构。
2. **LLM 提供什么**：冻结 LLM 的表示空间里天然包含概念关系（相似性/共现拓扑），可视作一个“语义监督空间/语义流形”。
3. **蒸馏蒸什么**：用结构化、受约束的 JSON token 目标去训练 ViT+projector，使视觉表征经映射后能在该语义空间里“落到正确位置”，从而获得更强迁移与更少 shortcut。

对应主线说明见：`00_方案总览.md`、`01_研究问题与贡献.md`。

## Q2：这是不是主故事线？会不会写成口号？

**A：是主故事线，但必须用“硬证据链”落地。**

你需要在实验里把价值钉死：
- **ViT-only transfer**：丢掉 LLM 后仍提升（external test / transfer / low-data）
- **anti-shortcut**：image-swap / prompt-corruption
- **可靠性**：risk–coverage + clinical hard errors

对应评测口径见：`07_实验矩阵与评估协议.md`。

## Q3：最终交付物是什么？是“更懂 LLM 的 ViT”吗？

**A：不是。主产物是一个更强的医学 ViT backbone checkpoint（+ 一套训练范式）。**

部署/推理时可以不需要 LLM：直接把 ViT 接分类/分割/回归/检索等下游头。

对应“交付物”写法见：`01_研究问题与贡献.md`、`00_方案总览.md`。

## Q4：ViT-only transfer 是什么意思？为什么它关键？

**A：评测时把 LLM 丢掉，只用 ViT encoder 做下游任务。**

它直接回答审稿人最关心的问题：提升到底来自 “LLM 在推理兜底”，还是 “ViT 表征真的变强”。

对应实验设计见：`07_实验矩阵与评估协议.md`。

## Q5：LLM swap 是什么意思？为什么要做？

**A：固定 ViT+projector，不改训练出来的视觉表征，替换 teacher LLM（例如 Llama ↔ Qwen）。**

目的：证明你学到的不是“某个 LLM 的癖好”，而是对语义监督空间的更通用对齐。

对应实验设计见：`07_实验矩阵与评估协议.md`。

## Q6：表里那些 `null` 字段怎么处理？会不会变成噪声？

**A：`null` 不是噪声，是“不可答”的正式答案；要闭环到训练与评测里。**

推荐做法：
- 编译阶段：字段缺失/不可答 → 显式写 `null`，并配套 `answerability=false`、`uncertainty` 与 `provenance`
- verifier 阶段：不允许省略 key；缺失 key 直接 `missing_field`
- 训练阶段：`answerable=false` 的字段目标固定为 `null/uncertain`，并走 abstain 分流；不进入 ranking 等“真假对比”

对应规范见：`05_UMS_与Verifier_与FailureTaxonomy.md`（含 `null` 规范、verifier 规则与 role 分流），训练口径见：`06_训练目标与SamplePolicy.md`。

## Q7：为什么不用现成 VLM 当主线？

**A：主线卖点是“LLM 是训练时的监督空间，ViT 是产物”。**

若直接复用医学 VLM 的视觉编码器，审稿人会质疑：提升到底来自你方法，还是来自“原模型已经学过医学数据”。因此主线建议自己搭最简 ViT+projector+frozen LLM，并把现成 VLM 放在 baseline 里做公平对照。

对应模型与 baseline 口径见：`04_模型.md`、`07_实验矩阵与评估协议.md`。

## Q8：05/06/07 分别负责什么？

**A：三个文件分别对应：规范、训练、评测（含可靠性）。**

- `05_UMS_与Verifier_与FailureTaxonomy.md`：UMS-JSON 字段语义、`null` 规范、deterministic compiler 原则、verifier 规则、failure taxonomy、role 分流与产出报告
- `06_训练目标与SamplePolicy.md`：损失函数（`L_tok/L_rank/L_vdep/L_ans`）与 verifier-guided sample policy（哪些样本进哪些 loss）
- `07_实验矩阵与评估协议.md`：主结果、baseline、公平性、关键“堵嘴实验”（ViT-only、LLM swap）、消融矩阵，以及 reliability/hard tests 套件

## Q9：和 ViTP（Visual Instruction Pretraining）是不是一类？怎么避免像复刻？

**A：属于同一大类“用高层语义信号塑造视觉骨干”，但你必须把差异做成硬分叉：**

1. **训练/交付**：LLM 冻结 + 只交付 ViT（ViT-only、LLM swap 作为证据）
2. **监督形态**：结构化可验证 JSON（UMS + constrained + verifier），而不是自由文本
3. **医学价值**：主指标强调可靠性（hard errors、risk–coverage）与跨域迁移，而不是“会不会回答”

对比写法见：`01_研究问题与贡献.md`、写作模板见：`09_写作大纲与图表清单.md`。

## Q10：novelty 会不会偏弱？看起来像“训练了一个 ViT 权重”？

**A：如果你把贡献写成“更强 ViT checkpoint”，novelty 会显得弱；但如果你写清楚“训练机制”，novelty 可以是中上。**

建议把贡献钉成三条“可复现、可消融”的方法贡献（而不是结果贡献）：

1. **Verifiable Semantic Distillation**：冻结 LLM，仅更新 ViT+projector，用结构化 token 对齐把语义结构蒸馏进视觉表征。
2. **Answerability-aware learning**：字段级 answerability/abstain（训练时 gating，推理时 risk–coverage），把不可答从噪声变成信号。
3. **Constraint-aware training + verifier-guided policy**：训练阶段 masked-CE 约束一致性 + verifier role 分流进入训练算法。

并用“三锤硬证据”落地：ViT-only transfer、anti-shortcut、hard errors + risk–coverage（见 `07_实验矩阵与评估协议.md`）。

## Q11：这是不是就是“多任务学习”？

**A：不是传统意义的多任务学习。**

- 传统多任务：一个 backbone + 多个任务头（分类/分割/回归），每个任务用各自 loss（CE/Dice/MSE），任务关系主要靠参数共享“隐式发生”。
- 我们的方法：把不同监督统一到 **UMS-JSON 的结构化 token 预测接口**（单一输出空间），并借助冻结 LLM 的语义结构 + 约束一致性 + verifier 分流，显式塑造表征与可靠性行为。

对应方法定义见：`02_方法.md`、`05_UMS_与Verifier_与FailureTaxonomy.md`。

## Q12：这个 ViT 到底怎么“用且有用”？接分类头/分割头/回归头还是接 LLM？

**A：默认是 ViT-only（更符合临床部署），ViT+LLM 是可选交互层。**

- 形态 A（主线）：**ViT-only + 轻量 head**（分类/分割/回归/检索）。主结果表也应以 ViT-only 为主。
- 形态 B（可选）：**ViT + Frozen LLM** 做结构化报告/交互，但不把它当主交付物，避免变成“又一个聊天模型”。

论文里证明“有用”的证据链建议固定为：external/transfer、low-data、可靠性（risk–coverage + hard errors）、部署成本（ViT-only 推理资源）（见 `07_实验矩阵与评估协议.md`）。

## Q13：如果 ViT+LLM 比不过 BiomedCLIP 这类图文预训练怎么办？

**A：别把战场选在“通用 VLM 能力/会不会回答”，而是选在你更有临床优势的维度上。**

你的主结论是：**训练期用冻结 LLM 做监督，部署期只用 ViT**；因此主表重点应放在 ViT-only 的 external/low-data/可靠性收益。

同时准备一个很稳的实验（强烈推荐）：  
**BiomedCLIP init + Ours**（用 BiomedCLIP encoder 初始化再跑你的训练机制），证明你贡献的是“训练机制”，不是“预训练数据规模”（见 `07_实验矩阵与评估协议.md`）。

## Q14：必须和“直接接分类/分割/回归头训练（task-head baseline）”对比吗？

**A：必须，而且建议放主表/主消融第一行。**

这是审稿人心中最公平的 baseline：同数据、同 backbone、同算力下，“为什么不直接训任务头？”  
你不比，很容易被判定为 busywork。

更现实的预期（写进论文就不怕纠缠）：
- 你未必在“同分布满数据单任务”处处赢 task-head baseline
- 但你应当在 external/low-data/可靠性（hard errors、risk–coverage、anti-shortcut）上更有优势（见 `07_实验矩阵与评估协议.md`）。

## Q15：ViTP（Visual Instruction Pretraining）用的是结构化数据吗？

**A：大部分是“指令-回答”的自然语言监督，结构化程度弱于 UMS-JSON。**

- ViTP 的基本形式可理解为 `(image, instruction/question, response)`，以 response tokens 作为 next-token 监督；部分下游任务会被“包装成 QA”，可视为半结构化文本。
- 医学部分可能包含 ROI/mask 相关的 VQA/bench，但并未将输出统一压到严格 schema（例如严格 JSON），也没有把程序化 verifier 做成训练闭环的核心组件。

更详细的对照笔记见：`01_研究问题与贡献.md`（6.4）。

## Q16：ViTP 原文怎么评测能力？也是训完拆 ViT，加各种任务头测试吗？

**A：是的，评测口径本质上也是“抽 backbone → 接下游头微调/迁移”。**

- 这点对你是利好：你主线的 **ViT-only transfer** 与 ViTP 的“拆 backbone 做下游”在 protocol 上可对齐（关键是 head/训练预算/数据划分要写死）。
- 对话笔记里记录 ViTP 在医学分割常用 `ViT-Adapter + segmentation head（如 UperNet）` 并报告 mDice（引用前需核对原文实现细节）。

对齐建议见：`07_实验矩阵与评估协议.md`。

## Q17：ViTP 相比 nnU-Net / MedMAE 的表现怎么样？

**A：对话笔记里记录其在若干医疗分割数据集上优于 nnU-Net 与 MedMAE，但数值需要你对照原文核对后再引用。**

对话笔记（占位，待核对原文）：
- AMOS2022（CT, mDice）：ViTP 90.60 vs nnU-Net 87.28 vs MedMAE 90.11
- BraTS2021（MRI, mDice）：ViTP 72.11 vs nnU-Net 71.03 vs MedMAE 69.59
- CovidQUEx（CXR, mDice）：ViTP 91.55 vs nnU-Net 90.41 vs MedMAE 90.18
