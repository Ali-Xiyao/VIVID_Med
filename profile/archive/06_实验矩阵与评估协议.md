# 实验矩阵与评估协议（MICCAI 口径）

## 1. 主结果必须回答的 3 个问题

1. **ViT-only 是否更强？**（LLM 是工具）
2. **跨域/低数据是否更稳？**（表征是可迁移的）
3. **可靠性是否真的改善？**（hard errors 与 risk–coverage）

## 1.1 “是不是瞎忙活”的分水岭：把战场选在医学痛点上

你不需要把论文写成“我们在同分布满数据上处处压一切 baseline”。医学场景更有说服力、也更公平的主战场是：

- **external test / transfer（跨医院/跨来源）**：训练 A、测试 B 的分布漂移是常态
- **low-data（1%/5%/10%）**：标注昂贵，少样本效率更贴近现实
- **可靠性**：允许 abstain 的 risk–coverage + clinical hard errors（laterality/unit/×10）比小幅 AUC 更“临床”

因此：即使 task-head baseline 在“同分布满数据单任务”上更强，你的主结论仍可成立——只要你在上述战场上给出硬证据链。

## 2. 评测任务（v1.0 最小集）

- **CXR：CheXpert → NIH external**：对齐共同标签子集，报告 AUC/F1（NIH 仅用于 external test）
- **CT：AMOS → KiTS21 transfer**：报告 Dice/HD95（或同类分割指标），并做 low-data 1%/5%/10% 曲线

可选加分项（占位）：
- 测量回归：LIDC-IDRI（MAE + unit/hard error）

## 3. Baselines（必须对齐“训练谁”）

### 3.1 ViT 表征预训 baselines

- ImageNet 初始化
- MAE
- DINO/DINOv2

### 3.2 医学 VLP/VLM baselines（按可复现程度选择）

- MedCLIP / BioViL / BiomedCLIP / MedKLIP（取 encoder 做同样下游）
- LLaVA-Med / RadFM（尽量抽 encoder 对齐评估口径）

强烈建议额外加一行（能显著降低“比不过大预训”的风险）：

- **BiomedCLIP init + Ours**：用 BiomedCLIP 的视觉 encoder 初始化，再跑一遍你的训练范式；对比 “BiomedCLIP encoder → 下游” vs “BiomedCLIP init + Ours → 下游”，证明你贡献的是**训练机制**而不是数据规模。

> 具体选择与版本号：待定（列在 `11_待确认与留空项.md`）。

## 4. 关键“堵嘴实验”（必须）

### 4.1 LLM swap（最能证明不是过拟合某个 LLM）

v1.0 推荐口径（工程可复现、论文也好解释）：

- **freeze ViT**（视觉表征固定），替换 LLM（例如 Llama↔Qwen；不强求 tokenizer 一致）
- 仅用少量数据（1k–5k）训练一个 `projector_swap` 去适配新 LLM 的 embedding space
- 目标：结构化生成质量与 ViT-only 迁移结果不“崩盘”，从而支持“视觉表征可复用、语言接口可插拔（通过小适配器）”

### 4.2 ViT-only transfer（核心结论：ViT 是产品）

丢掉 LLM，仅用 ViT encoder 接 head/下游框架，比较：
- baseline ViT（ImageNet/MAE/DINO/医学预训）
- ViTP（如可对齐：抽 backbone → 接标准下游头微调；实现细节与 head 选择需对照原文核对）
- 我们的方法（VIVID-Med）

## 5. 消融（形成因果链，至少 6 行）

建议至少 8 行（和你“最终版”方案一致）：

1. **task-head baseline（必须）**：同样 UMS 数据，ViT + 任务头（分类/分割/回归）
2. **Token baseline**：仅 `L_tok`（普通 CE，训练不加约束）
3. + structured targets（UMS + deterministic compiler）
4. + verifier policy（role 分流/降权/负样本）
5. + `L_rank`（规则级 negatives）
6. negatives：规则级 → 规则+语义 → 规则+语义+在线挖掘（至少做两档）
7. + `L_vdep`（anti-shortcut）
8. + `L_ans`（answerability head）
9. constrained training：train w/o constraint vs train w/ masked-CE（推理都 constrained）
10. Frozen vs Unfrozen LLM（堵嘴项；如果做了 9 行太多，可把这一行放附录）
11. （可选但很加分）**teacher capacity ablation**：小/中/大 LLM（或不同容量配置），验证存在 sweet spot（teacher 太强会“替学生做题”，ViT 学不到）。

### 5.1 Plan A / Plan B（写进论文就不怕 reviewer 纠缠）

- **Plan A（理想）**：你在 external/low-data/可靠性上显著优于 task-head baseline，同时同分布结果不落后太多 → 主线最强。
- **Plan B（常见且可接受）**：task-head baseline 在同分布满数据上更强，但你在 external/low-data/硬错误/拒答可靠性上更强 → 论文主张写成“面向临床部署的鲁棒与安全表征”，主线依然成立。

## 6. 公平性与成本对齐（写作必备）

- 相同 backbone、相近训练步数/epoch、相同数据规模（或明确说明差异）
- 报告 compute：GPU 型号、训练时长、显存、是否量化/是否 checkpointing
- 统一评估口径：VLM baseline 取 encoder 后的下游设置要尽可能一致

## 7. 结果呈现建议（主文优先展示什么）

- 主结果表必须是 **ViT-only**（丢 LLM）在 CheXpert→NIH 与 AMOS→KiTS21 上的结果
- reliability 主图（risk–coverage + hard error breakdown）尽量放主文
- LLM swap 放主文（哪怕只展示关键字段/关键任务），用来一锤定音

## 8. 可靠性与 Hard Errors 评测套件（v1.0）

> 本节与“实验矩阵/消融/公平性”合并在同一文件，确保“主结果 + 可靠性证据链”在同一处。

### 8.1 为什么必须单独做“可靠性主表/主图”

医学审稿人常问的不是“平均指标提升多少”，而是：
- 错在什么类型？（laterality、单位、测量、漏小病灶…）
- 什么时候会错？能不能拒答？
- 在给定 coverage 下，hard errors 能否显著降低？

### 8.2 指标（v1.0 最小集）

#### 8.2.1 格式正确性（必须显式报告）

- **schema violation rate**：应接近 0（训练+推理都 constrained 时更容易做到）
- **illegal enum / illegal number format rate**：可并入 schema violation 或单列

#### 8.2.2 校准（Calibration）

- ECE（Expected Calibration Error）
- Brier score

#### 8.2.3 风险-覆盖（Risk–Coverage / Selective Prediction）

输出一个置信度/风险分数（建议与 answerability/abstain 打通），绘制：
- coverage vs error（或 vs hard error）
- hard error rate @ fixed coverage（例如 90% coverage 时 laterality_error）

#### 8.2.4 Clinical hard errors（建议最少包含）

- `laterality_error`（左/右/双侧判错；注意：很多样本应为 unanswerable → 应评估 abstain）
- `unit_error`（mm↔cm、面积/体积单位不一致）
- `measurement_hard_error`（超过阈值的绝对误差/相对误差；阈值待定）

> v1.0 建议把 laterality/view 的“不可判定”单独建模：用 `answerability head` 预测并触发 abstain，从而在 risk–coverage 下显著降低 hard errors。

> hard error 的阈值与字段集合：见 `11_待确认与留空项.md`。

### 8.3 anti-shortcut 硬测试（建议固化为 test suite）

#### 8.3.1 Prompt corruption

对 prompt 做同语义扰动（删词/同义改写/模板变化），要求结构化输出不变（或变化在可解释范围内）。

#### 8.3.2 Image swap

固定 prompt，交换图像，要求结构化输出发生变化；并统计 swap sensitivity。

#### 8.3.3 Hard-negative stress test

对同一图像提供“真值 JSON vs 格式合法假 JSON”，评估模型排序/偏好能力（与 `L_rank` 对齐）。

### 8.4 输出产物（建议固定）

- `reports/reliability_main.csv`：ECE/Brier、coverage@risk、hard error rate@coverage
- `figs/risk_coverage.png`：risk–coverage 曲线（主文优先）
- `figs/hard_error_breakdown.png`：hard error 类型分解
- `reports/prompt_corruption_sensitivity.csv`
- `reports/image_swap_sensitivity.csv`
