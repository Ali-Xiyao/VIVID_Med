## 5. Baseline 训练方式（详细口径）
以下是当前 baseline 的完整训练定义，用于和 VIVID 公平对比：

### 5.1 配置与脚本
- 配置：`configs/baseline_vit_chexpert_top7_dense10k.yaml`
- 训练脚本：`scripts/train_vit_baseline.py`
- 评估脚本：`scripts/eval_vit_baseline.py`

### 5.2 数据口径
- 数据源：`chexpert_ums_train.jsonl` / `chexpert_ums_val.jsonl`
- 标签集合：固定 `Top7` 字段（与 V6 一致）
- 采样策略：`train_dense_top_k=10000`，`val_dense_top_k=1000`
- 不确定标签策略：`uncertain_policy=ignore`（训练与评估都可忽略 -1/NaN）

### 5.3 模型结构
- 主干：`vit_base_patch16_224`
- 初始化：`vit_pretrained: true`（ImageNet 预训练，不是从零开始）
- 头部：直接多标签分类头（输出 7 维 logits）
- 与 VIVID 差异：baseline 不经过 `LLM`、不做 JSON 生成、不用 prompt。

### 5.4 损失函数与优化
- 损失：`BCEWithLogitsLoss(reduction="none") + mask`
- 只对可监督位置计算 loss（NaN/忽略位不反传）
- 优化器：`AdamW`
- 学习率策略：`Linear Warmup + CosineAnnealing`
- 典型训练参数：
- `lr=1e-4`
- `batch_size=16`
- `grad_acc=2`
- `max_steps=4500`（Top7 dense10k 配置）

### 5.5 评估方式
- 阈值：`0.5`
- 输出：`macro_f1`、`micro_f1`、per-label `precision/recall/f1`
- 重点：baseline 是“纯分类”任务；VIVID 是“生成+解析后的分类评估”，两者任务形式不同，解释时需分开讲。

### 5.6 与 VIVID 的公平对比原则
- 同字段集（Top7）
- 同训练/验证子集（dense10k / val1k）
- 同 uncertain 策略口径（至少并列报告）
- 同评估脚本口径（是否允许 abstain 要明确写在表头）

## 10. 是否超过 baseline（当前结论）

- 指标：`macro_f1=0.9125`、`micro_f1=0.8948`
- 性质：纯 ViT 分类器（直接多标签 BCE），不是生成任务。

### 10.2 当前是否超越
- **没有超越。**
- 以 `baseline_vit_v2` 的 Top7 字段估算参照（由 per-label 汇总）：`macro_f1≈0.8882`、`micro_f1≈0.9006`。
- V6 主线（Top7 全覆盖口径）显著低于 baseline 上界。
- V5.1 Top3 虽明显提升，但仍未稳定超过 baseline 的同类头部字段表现。

### 10.3 为什么没有超过（实质原因，不是一句“数据少”）
1. 监督路径效率差异：baseline 直接分类监督；VIVID 是“生成再解析”，同样样本下有效梯度更稀。
2. 字段不均衡：`blank/NaN` 背景下，部分字段可答但正样本召回难，模型趋向保守输出。
3. 目标冲突：生成任务既要结构正确又要状态正确，结构 token 与状态 token 共享学习预算。
4. 口径影响：`abstain` 会提高可评估样本上的分数，但覆盖下降后不代表真实全量能力。
5. 标签难度差异：`Pleural Effusion` 这类字段已暴露“高精度低召回”，说明主要缺陷是漏检。

## 16. 为什么 V7.1 比 V7 差（关键诊断）

### 16.1 现象
- V7（GPU0）：`macro_f1=0.5115`，`micro_f1=0.7051`
- V7.1（GPU1）：`macro_f1=0.4617`，`micro_f1=0.6686`
- 差值：`macro -0.0498`，`micro -0.0365`

### 16.2 直接原因（配置层）
- `field_query_training` 更激进：
  - V7：`k_min=2, k_max=3, focus_probability=0.8`
  - V7.1：`k_min=3, k_max=3, focus_probability=1.0`
- 标签权重更强：
  - V7.1 提高了 `present` 及低召回标签的权重（如 `Pleural Effusion/Edema/Pneumothorax`）。
- 结果是训练分布偏离全量评估分布，导致“局部强化”但整体泛化下降。

### 16.3 训练动态证据
- V7.1 在 `step 400` 达到最好 `val_loss=0.1879`，之后长期变差到 `step 3000 -> 0.2271`。
- 说明 V7.1 是“过早最优 + 后期退化”，不是稳定增益路线。

### 16.4 解释
- V7.1 的改动方向本意是修低召回，但强度过高，导致模型在部分标签上“策略性过拟合/误判”，把总体 macro/micro 拉低。
- 所以 V7.1 不是无效思路，而是当前参数过激、未与评估分布对齐。

## 17. 全项目审查：问题诊断与 MICCAI 投稿改进路线（2026-02-12）

### 17.1 核心问题诊断（6 项）

#### 问题 1：主线实验结果远低于 baseline，且差距没有收敛趋势
- V7 主线 `macro_f1=0.5115` vs baseline `macro_f1=0.8839`，差距 42%。
- 从 V4→V6→V7 的迭代来看，macro_f1 的提升速度在放缓（V4.2: 0.2553 → V6: 0.4087 → V7: 0.5115）。
- 按这个趋势继续调参很难追上 baseline。
- **结论**：当前的"生成链路直接对比分类链路"这条路走不通，不能作为 MICCAI 主表。

#### 问题 2："同头公平对比"是唯一正面结果，但增益太小
- From-VIVID vs Baseline: `macro_f1 +0.0018`, `micro_f1 +0.0011`。
- 没有多次重复实验的标准差，reviewer 会直接质疑这是随机波动。
- **结论**：需要多 seed 重复 + 统计检验才能让这个结论站住。

#### 问题 3：实验矩阵严重不完整（对标 MICCAI 要求）
- 文档 07 规划了完整的实验矩阵，但目前实际完成的只有 CheXpert 上的 Top7 分类（单数据集、单任务）。
- 缺失项：
  - NIH external test（跨域迁移）
  - CT 数据（AMOS/KiTS21）
  - Low-data 实验（1%/5%/10%）
  - LLM swap 实验
  - Anti-shortcut 测试（prompt corruption / image swap）
  - 可靠性指标（ECE/Brier/risk-coverage）
  - 消融实验（至少 6 行）

#### 问题 4：评估协议不够严谨
- 只有 1 次运行，没有多 seed 重复（MICCAI 通常要求 3 runs + std）。
- 没有 AUC 指标（医学影像标配）。
- `compute_risk_coverage_curve` 和 `compute_clinical_hard_errors` 已实现但从未被调用。
- 没有统计显著性检验。

#### 问题 5：训练策略存在过度工程化风险
- V7 配置中同时启用了 `field_query_training` + `token_weighting` + `answerability_mask` + `label-specific weights`，变量太多。
- V7.1 的退化已经证明了这一点——参数空间太大，调参方向不明确。

#### 问题 6：代码层面的问题
- `RankingLoss`、`VisualDependenceLoss` 已定义但 `lambda_rank=0.0`, `lambda_vdep=0.0`，从未实际使用。
- `AnswerabilityLoss` 同样 `lambda_ans=0.0`，未启用。
- `generate()` 方法中的 JSON 提取逻辑是 heuristic 的，没有用到文档中描述的 constrained decoding。
- 没有实现文档中提到的 hard negative 构造。

### 17.2 策略转向建议

**核心判断**：放弃"生成链路直接超越 baseline"，聚焦 ViT-only transfer。

文档 01 和 00 已经写得很清楚——"LLM 是工具，ViT 是产品"。当前的主要错误是把精力花在让生成链路的 F1 追上分类 baseline，但这不是论文的主张。正确的主战场是：

> 用 VIVID 预训练的 ViT backbone，在下游任务上（尤其是跨域和低数据场景）超过 ImageNet/MAE/DINO 初始化的 ViT。

写作策略按文档 00 中的 Plan B：不追求在同分布满数据上压过 task-head baseline，而是把主张定位在"面向临床部署的鲁棒与安全表征"——在 external transfer、low-data、可靠性上给出硬证据。

### 17.3 执行优先级

#### P0（必须完成，否则无法投稿）
1. **同头对比做扎实**：From-VIVID vs Baseline 做 3 个 seed 重复，报告 mean±std，做配对 t-test。
2. **NIH external test**：用 CheXpert 训练的 ViT backbone 迁移到 NIH，对比 ImageNet init vs VIVID init。
3. **Low-data 实验**：在 CheXpert 上做 1%/5%/10% 标注量的 linear probe / finetune，画学习曲线。
4. **消融实验**：至少 4-6 行（token weighting on/off, answerability mask on/off, constrained vs unconstrained, Top3 vs Top7 vs Top14）。

#### P1（强烈建议，显著提升录用概率）
5. **LLM swap**：换一个 LLM（如 Qwen2.5-0.5B 或 Qwen2.5-3B），证明 ViT 表征不依赖特定 teacher。
6. **可靠性指标**：ECE、risk-coverage 曲线，利用已有的 `evaluation/metrics.py` 代码。
7. **更多 baseline**：至少加 MAE 和 DINO/DINOv2 的 ViT 初始化对比。
8. **实现 constrained decoding**：文档中描述了但代码未实现。

#### P2（加分项）
9. CT 数据实验（AMOS→KiTS21），证明方法不限于 CXR。
10. Anti-shortcut 测试（image swap / prompt corruption）。
11. Hard negative ranking loss 的实际启用和消融。

### 17.4 当前状态总结
- 项目设计文档质量高，方法论思路清晰，但实验执行和文档规划之间有很大的 gap。
- 最紧迫的是停止在生成链路 F1 上继续调参，转向 ViT-only transfer 的系统性实验。
- 把 From-VIVID 的微弱优势通过跨域/低数据/多 seed 放大为可信的结论。

## 19. V8 设计与执行计划（2026-02-12）

### 19.1 问题诊断（为什么 From-VIVID 只比 baseline 高 +0.0018）

- ViT 只输出 1 个 CLS token（768 维），Projector（~7M 参数）足以独立拟合 LLM 输入，ViT 几乎不需要改变
- ViT 和 Projector 共用 lr=8e-5，Projector 优先收敛，ViT 梯度被稀释
- 训练步数不足（3500 步，有效训练量受 ~50% null label 限制）

### 19.2 V8 核心改动

| 项目 | V7 | V8 | 理由 |
| --- | --- | --- | --- |
| vit_output_type | cls (1 token) | all (197 tokens) | 迫使 ViT 在 patch 级别编码信息 |
| num_prefix_tokens | 16 | 4 | 减少 Projector 免费容量 |
| projector_mlp_hidden_dim | 3072 | 1536 | 削弱 Projector |
| 学习率 | 统一 8e-5 | ViT=2e-5, Proj=1e-4 | 差异化学习率 |
| max_steps | 3500 | 5000 | 更多有效梯度 |

- VIVID 预训练：`configs/cxr_chexpert_v8_top7_gpu0.yaml`
- Transfer 评估：`configs/baseline_vit_chexpert_top7_dense10k_from_vivid_v8.yaml`（backbone_lr=1e-5, head_lr=1e-3）
