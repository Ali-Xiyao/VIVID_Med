# 研究问题与贡献（VIVID-Med v1.0）

## 1. 医学场景的结构性难点（审稿人最在意的 3 点）

1. **监督不可验证**：自然语言答案无法被程序化验真；临床关键字段（laterality、单位、测量值）出错代价高。
2. **语言捷径（shortcut）**：模型可能学会“读 prompt/先验”而不是“看图”，导致看似正确但不可信。
3. **你到底训练了什么**：如果 LLM 很强，提升可能来自 LLM 兜底而不是 ViT 表征变强（部署时不成立）。

## 2. 核心研究问题（Primary Research Question）

**在不训练（或尽量少训练）LLM 的前提下，能否用冻结 LLM 作为“结构化语义解码器/监督空间”，把可验证的结构化监督信号传导给 ViT，使 ViT 学到更可迁移、更可靠的医学视觉表征？**

## 3. 次级问题（Secondary Questions）

- 如何把公开医学数据集的 label/measurement/metadata 统一成一个可审计的结构化答案（UMS-JSON），并显式处理字段缺失与不可判定？
- 如何把 verifier 从“预处理清洗”升级为“训练算法的一部分”（样本分流/降权/负样本），从而形成可归因的新颖性？
- 如何用硬测试与硬指标证明模型“真的看图”，并把“知道什么时候不能答”纳入可靠性评测？

## 4. 论文主张（你要“钉死”的中心论点）

1. **LLM 是工具，ViT 是产品**：训练期间利用 LLM 的 token 空间作为监督接口；训练后丢掉 LLM，ViT-only 在下游仍显著提升。
2. **可验证结构化监督是必要的**：输出空间被压到 schema 允许的 token 集合 + verifier 单元测试，正面回应“幻觉/不可追责”质疑。
3. **anti-shortcut 必须可量化**：prompt-corruption / image-swap / risk–coverage + clinical hard errors 的组合证据链必须成立。

## 4.1 Novelty 抓手：Semantic Knowledge Distillation（建议放在 Intro 第一段）

传统 one-hot 标签默认“疾病彼此正交”；而冻结 LLM 的 token 分布/语义空间天然编码医学概念的相似性与共现拓扑。本方法的核心思想是：用**受约束的结构化 token 监督**把视觉表征“蒸馏”到该语义流形上，从而在跨域与低数据场景更稳。

把这句话拆开理解（建议写进你自己的“讲稿/答辩备注”，写论文时可择要用）：

1. **one-hot 的问题**：分类训练把“肺炎/实变/积液”等标签当作彼此独立的维度，监督信号只强调“对/错”，不鼓励模型显式学到概念相似性与共现结构。
2. **LLM 语义空间提供什么**：在冻结 LLM 的表示空间里，“pneumonia” 与 “consolidation” 这类概念往往更接近、共现关系更强——这是一种天然的“语义拓扑/语义流形”结构。
3. **蒸馏蒸的是什么**：我们不让 ViT 学 one-hot 边界，而是让它在“结构化、受约束”的 JSON token 监督下，为了让冻结 LLM 输出正确字段，反向塑造 ViT 表征，使其经 projector 映射后能落在 LLM 语义空间的“正确位置”。

> 这段就是论文主故事线：**LLM 是训练时的老师（监督空间），ViT 是最终交付物（产物）**。

## 4.2 最终交付物是什么（一定要写清楚，避免“瞎忙活”观感）

你最后交付的不是“更懂 LLM 的 ViT”，而是：

- **产物 A（主产物）**：一个更强的 **medical ViT backbone checkpoint**（以及对应训练范式/配方）。部署/推理时可以**完全不使用 LLM**，直接把 ViT 接到分类/分割/回归/检索等下游头上。
- **产物 B（可选加分）**：一个不含 LLM 的轻量结构化输出头（或小 decoder），让 ViT 直接输出 UMS 字段（仍配合 verifier 做 safety check）。这部分可做附录/后续版本。

审稿人要的不是“我们用没用 LLM”，而是你能不能给出三锤硬证据（见 `07_实验矩阵与评估协议.md`）：
- **ViT-only transfer**：丢掉 LLM 后，external test / transfer / low-data 仍显著提升
- **anti-shortcut**：image-swap / prompt-corruption 证明输出依赖视觉证据
- **临床硬错误 + 风险覆盖**：laterality/unit/×10 等 hard errors 下降，risk–coverage 更安全

## 5. 核心贡献（建议写在 Abstract/Intro 的 3–4 点）

> v1.0 建议写 3 条主贡献 + 1 条评测协议（可选但很加分）。

1. **LLM-mediated Structured Alignment**  
   冻结 LLM，仅更新 ViT+projector，通过 next-token prediction 学习生成满足 UMS-JSON 的结构化 tokens。
2. **Verified Supervision：UMS + Deterministic Compiler + Programmatic Verifier**  
   统一答案格式（UMS-JSON）；尽可能用规则/几何/元数据确定性计算真值；verifier 输出 failure types 与训练角色（role）。
3. **Verifier-guided Training Policy（不是清洗，而是训练接口）**  
   把 `pass/fail/failure_type/confidence` 映射为 `positive/abstain/negative-only/drop`，决定样本进入哪些 loss（形成可消融的因果链）。
4. **Hard Reliability Protocol（可选）**  
   固化“临床硬错误 + 风险覆盖”的评测协议：hard error rate@coverage、ECE/Brier、image-swap/prompt-corruption 套件。

> v1.0 最终版建议把 “Answerability head（可答性预测 + 推理 gating）”写进贡献 2 或 3（它能把 laterality/view 的不可判定从噪声升级为训练信号）。

## 6. 与 ViTP 的关系（Related Work/Discussion 口径）

### 6.1 ViTP 的新颖性“够吗”？

ViTP 在“提出一个新预训练范式（用 instruction-following 反向训练视觉骨干）+ 一个简单有效正则（大比例视觉 token drop/VRL）”意义上，novelty 是成立的；但在 MICCAI 严格审稿里，它更偏**中等偏上**：容易被认为是 VLM/SFT 思路的自然延伸，VRL 也可能被类比为 dropout/masking 的变体。

### 6.2 VIVID-Med 要站住的差异点（建议写成 3 句）

- 我们把“能生成答案”降维为“生成可验证的结构化答案”，并把 failure taxonomy 与 abstain 机制纳入训练与评测。
- verifier 不只是清洗器，而是进入训练算法（role/weight/negatives）的接口，可做因果链消融归因。
- 我们提供可复现的 anti-shortcut 硬测试与 hard error@coverage 指标，证明模型依赖视觉证据而非语言捷径。

### 6.3 你要避免“像复刻”的 3 个硬分叉（更像审稿人对照表）

即便同属“用高层语义/指令监督塑造视觉骨干”的大类范式，我们要明确站在一个更医学、更可验证的分支上：

1. **训练谁 / 交付谁**：LLM 全程冻结，只更新 ViT+projector；并用 **LLM swap + ViT-only transfer** 证明 LLM 只是工具、ViT 才是产物。
2. **监督长什么样**：监督目标是 **可验证的结构化 JSON（UMS）**，训练/推理都走约束一致性（masked-CE + constrained decoding）并接入 verifier 单元测试，而不是自由文本指令响应。
3. **卖点是什么**：主卖点从“能做 VQA/指令跟随”转为 **医学可靠性**（answerability/abstain、clinical hard errors、risk–coverage）与跨域迁移表征。

### 6.4 ViTP 的训练数据与监督形态（对照笔记，引用前请核对原文）

> 下述为“对话整理笔记”，用于写作定位与实验对齐；最终写进论文前请对照 ViTP 原文逐句核对（尤其是数据配方/评测设置/数值）。

- ViTP 的基础训练样本可以理解为三元组：**(image, instruction/question, response)**，训练目标是对 `response tokens` 做 next-token prediction（指令跟随式监督）。
- “结构化”的部分主要体现在：作者会把部分下游任务的标签/标注**包装成问答形式**（可称为“半结构化文本”）；但其输出仍以自然语言 response 为主。
- 医学部分包含一些与 ROI/segmentation 相关的 VQA/bench（即输入/问题显式涉及 ROI 或 mask），但并未把所有任务统一压到一个严格 schema（如严格 JSON），也没有把程序化 verifier 作为训练闭环的一部分。

### 6.5 ViTP 的评测方式：预训练后抽 backbone → 接下游头微调（对齐口径）

- ViTP 的下游评测核心流程与“ViT-only transfer”一致：**预训练完成后抽取 ViT backbone**，再按任务接标准 head（分割/检测等）进行 finetune/transfer。
- 对医学语义分割，笔记里记录其常见做法是 **ViT-Adapter + segmentation head（如 UperNet）**，指标报告 mDice（具体实现与超参需核对原文）。

> 这也意味着：如果你把 ViTP 纳入 baseline，你的主结果表（ViT-only）在 protocol 上是可对齐的——关键在于把 head/训练预算/数据划分写死（见 `07_实验矩阵与评估协议.md`）。

### 6.7 我们相对 ViTP 的“优势表述”怎么写得更像 MICCAI

即使同属“用语言信号塑造视觉骨干”的大类，VIVID-Med 的优势应当被写成**医学系统性收益**而非“又一个预训练技巧”：

- **更可审计**：输出空间是 UMS-JSON + 训练/推理约束一致性 + verifier；失败类型可统计、可归因、可做训练分流消融。
- **更医学化的可靠性**：把 answerability/abstain、clinical hard errors、risk–coverage 作为主指标，避免只刷平均分。
- **更贴近部署**：LLM 训练期出现、部署期移除（ViT-only），并用 LLM swap 证明不依赖特定 teacher。
