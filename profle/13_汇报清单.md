# 13 汇报清单（明日可直接使用）

## A. 30 秒结论（开场）
- 核心方法未改：仍是“冻结 LLM + 训练 ViT/Projector”。
- 当前问题不在“能否解析 JSON”，而在“状态分类能力不足”。
- V3 已完成代码落地（未开跑），重点是 **state-aware token weighting** 抗 `null` 塌缩。

## B. 必报指标（1 张表）
建议直接展示以下 4 组：
1. `outputs/cxr_chexpert/checkpoints/best.metrics.full.json`
2. `outputs/cxr_chexpert_v2/checkpoints/best.metrics.full.v2.json`
3. `outputs/cxr_chexpert_quick_rerun/checkpoints/best.metrics.quick.json`
4. `outputs/baseline_vit_v2/best.metrics.json`

建议字段：
- `json_success_rate`
- `pred_nan_rate`
- `macro_f1`
- `micro_f1`

## C. 问题归因（1 页）
1. `null` 占比过高导致监督失衡。  
2. 固定 JSON 模板 token 过多，关键状态 token 梯度被稀释。  
3. 因此出现“格式正确但预测弱”的现象。

## D. V3 改动清单（代码级）
- `training/losses.py`：`StructuredLoss` 支持 token/sequence 加权。
- `training/trainer.py`：接入 `token_weighting`，训练和验证统一走加权 loss。
- `models/vivid_model.py`：`forward` 返回 `labels`，供加权损失计算。
- `scripts/train_cxr.py`：接入 `token_weighting` 参数，debug 自动禁用。
- `configs/cxr_chexpert_v3.yaml`：新增可直接训练的 v3 配置。

## E. 风险与对策（评委常问）
- 风险：加权可能引入某些标签过拟合。  
  对策：先看 `macro_f1` 与每标签 F1，再调 `present/uncertain/null` 权重比。  
- 风险：仅靠 token weighting 不足以追平 baseline。  
  对策：下一步预留“字段级辅助损失”方案。

## F. 汇报后执行计划（可落地）
1. 跑 `v3` 完整训练（不改数据划分）。  
2. 跑 full eval，和 `baseline_vit_v2` 同口径对比。  
3. 记录 per-label 改善与失败标签，进入下一轮权重搜索。
