## 3. 问题定位（当前结论）
1. **状态词监督不平衡**：训练目标里 `null` 比例高，导致模型倾向保守输出。  
2. **LM loss 稀释**：固定 JSON 模板 token 太多，关键状态 token（present/absent/uncertain/null）梯度占比不足。  
3. **现象一致**：解析成功不等于判别有效，当前瓶颈是“可解析”到“可判别”的迁移。

### 3.1 CheXpert findings 的 blank/NaN ≈ 70%（不是读取 bug）
- **结论**：当前大量 `null` 来自 CheXpert 原始标注的 `blank`（在 CSV 里表现为 `NaN`），语义是“报告未提及该 observation”，不是我们没读到标注。
- **代码口径（显式设计）**
- `data/dataset/prepare_chexpert.py:70`：`NaN -> state=None`（写入 JSON 即 `null`）。
- `data/dataset/prepare_chexpert.py:143`：`answerability[label] = not isna(raw)`，所以 `blank/NaN` 会变成 `answerability=false`。
- `data/chexpert_dataset.py`：训练标签向量里 `state=null -> label=NaN`（评估时对 `y_true` 的 NaN 位置做 mask）。
- 验证：在 `data/dataset/processed/chexpert_ums.jsonl` 中，`state=null` 的 finding 其 `answerability=false` 对齐率为 100%（说明这是数据语义，不是解析/读取错误）。
- **统计证据（30k 采样）**
- `data/dataset/processed/chexpert_sampled_30k.csv`：14 个 findings 总体 `NaN(blank)` 占比约 **70.47%**（与 UMS JSONL 中 `state=null` 占比一致）。
- Top missing（30k）：`Pleural Other 92.45%`、`No Finding 91.50%`、`Fracture 89.14%`、`Lung Lesion 88.82%`、`Pneumonia 83.63%`。
- 完整逐标签表见：`profle/schema/coverage_table.md:25`（present/absent/uncertain/missing/覆盖率）。
- **关键影响**：`configs/cxr_chexpert.yaml` 的 prompt 强制 “findings 包含全部 14 条”，但真实监督里大部分是 `blank->null`，导致 token 监督极端稀释，模型倾向学到“安全输出 null”，从而出现 `json_success_rate` 高但 `macro_f1` 低的现象。

### 3.2 缺失率怎么处理（从不改 UMS 到伪标签/字段重构）
- **对比口径（必须写清楚）**
- 严格口径（UMS 语义）：`blank/NaN -> unknown`，在 `y_true` 中为 NaN 并在评估时 mask 掉（当前实现）。
- baseline 口径（公平对比）：额外报告一套把 `blank->absent` 的指标（仅用于对比，不等价于 UMS 真值）。
- **路线 A：训练目标层面（不改变 UMS 语义）**
- 已落地：V3 `state-aware token weighting`（抑制 `null` token 主导）。
- 下一步（推荐）：`answerability-aware loss`，对 `answerable=false` 的 finding block 整段降权/屏蔽，而不是只给 `null` 字符串降权。
- 备选：增加轻量多标签分类辅助头（只更新 ViT/Projector），给 `present/absent/uncertain` 提供更密的梯度，与 JSON 生成联合训练。
- **路线 B：字段/输出设计回顾（减少“必须显式输出 14 个 null”的结构性问题）**
- 备选 schema：把 findings 从“全量字典 + 大量 null”改成“稀疏列表/集合表示”，例如 `{"present":[...],"uncertain":[...],"absent":[...]}`；未出现即隐式 unknown。
- 好处：显著减少模板 token 与 `null` token；更贴合稀疏监督。
- 风险：需要同步调整 verifier/eval 口径（可通过后处理还原为 14 label JSON 保持兼容）。
- **路线 C：数据层面稠密化（teacher 伪标签 + 小规模人工标注）**
- 伪标签：用 teacher（例如 ViT baseline，按常见 CheXpert 口径 `blank->absent`）对 `blank` 位置做高置信度填充（只填最确定的 `absent/present`，其余仍保留 null）。
- 人工标注（Codex 做工具）：构建一个 calibration set（建议 200–500 张，重点覆盖高 missing 标签），用于校准伪标签阈值并验证“blank 更像 absent 还是 unknown”。
- 标注产物：`annotation_queue.csv`（待标注清单）+ `manual_labels.csv`（人工结果，写入 `.gitignore`，不入库）。
- 采样建议：对 `Pleural Other/Fracture/Lung Lesion/Pneumonia` 各抽 50–100 张；其中 20% 双人复标用于一致性评估。
- 标注字段：`sample_id/original_path/label_name/state(present|absent|uncertain|skip)/annotator/timestamp/notes`。

## 6. V4 定义：Answerability Masked Training（待训练）
### 6.1 V4 的核心定义（和 V3 的区别）
- V3：主要是 `state-aware token weighting`，通过调高 `present/absent/uncertain` 权重来抑制 `null` 塌缩。
- V4：在 V3 基础上，增加 **answerability 掩码**，对 `answerability=false` 的 findings token 直接做 loss 降权或屏蔽（mask）。
- 目标：不改变 UMS 语义（`blank -> unknown/null`），但避免 70% 不可答字段主导梯度。

### 6.2 为什么把它作为 V4
- 当前瓶颈不是 JSON 生成失败，而是状态判别被稀释（`json_success_rate` 高但 `macro_f1` 低）。
- 仅靠 token weighting 仍可能被“结构模板 token”稀释；V4 进一步把不可答字段从主训练信号中移走。
- 相比“全量人工补标”，V4 成本更低、可先验证是否能快速提升分类有效监督。

### 6.3 V4 的训练口径（必须固定）
- **严格 UMS 口径（主结果）**：`blank/NaN -> unknown`，评估时对 `y_true` 的 NaN 位置做 mask。
- **baseline 对比口径（附录结果）**：额外报告 `blank->absent` 指标，仅用于和常见分类基线公平对比。
- 两套口径要并列报告，防止“指标看起来提升但语义口径变化”造成误判。

### 6.4 V4 的工程落地内容（本次未开跑）
- 训练损失：在 `StructuredLoss`/trainer 中引入 answerability mask 的 token 级权重。
- 配置项：新增 `answerability_mask.enabled / false_weight / true_weight / scope`。
- 实验对比：至少给出 `V3` vs `V4` 的同划分对比（macro/micro F1、pred_nan_rate、per-label F1）。

### 6.5 V4 配套数据策略（与训练一起执行）
- teacher 伪标签：只对 `blank` 位置做高置信度填充，低置信度仍保持 unknown。
- calibration 人工标注：抽 200-500 张高 missing 标签样本，用于校准伪标签阈值。
- 标注产物：`annotation_queue.csv` + `manual_labels.csv`（写入 `.gitignore`，不入库）。

### 6.6 V4 成功判据（汇报时可直接用）
- 保持 `json_success_rate` 不下降（或下降不明显）。
- `macro_f1` 相对 V3 稳定提升，且 `pred_nan_rate` 明显下降。
- 至少 4 个高 missing 标签中，有 2 个以上标签 F1 出现可复现提升。
