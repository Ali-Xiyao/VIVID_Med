# 数据集

> 本文件聚焦数据集选择、字段覆盖、规模与合规；模型默认配置见 `04_模型.md`。
>
> 目标：在不引入“全新 3D 方案复杂度”的前提下，用 **≥3 数据集 + 明确 external test / transfer** 把泛化证据做实。

## 1. 数据集组合（可录用组合）

### 1.0 数据集选择要求（用于扩展/替换数据集时的 checklist）

> 目的：让你后续换数据集时不靠“感觉”，而是对齐到 UMS 字段与论文证据链（external/low-data/可靠性）。

**必须（Must）**

- **公开可获取 + 合规可写**：明确数据 origin/license/DUA（能写进论文 Data Availability / Ethics；本文件末尾也给了可直接复用的合规写法）
- **至少提供一种可验证监督**（三选一即可，越多越好）：
  - `findings`：病灶/所见标签（分类信号）
  - `geometry`：mask/bbox/keypoints（定位/分割信号）
  - `measurements`：带 unit 的数值（最好可由 geometry+spacing 确定性计算/校验）
- **必须有泛化证据场景**：至少包含一个 external test 或 transfer（跨机构/跨来源/跨任务），否则很难支撑“临床泛化”主张
- **允许 coverage 不全，但必须可表达**：缺失/不可答字段要能在 UMS 中显式写为 `null` + `answerability=false`，并通过 verifier（详见 `05_UMS_与Verifier_与FailureTaxonomy.md`）

**建议（Should）**

- 有可靠的 DICOM spacing 或 dataset metadata（便于 measurement 的确定性计算与校验）
- 有 uncertain/ambiguous 标签或可推导不可答字段（更适合 answerability/abstain 闭环）
- 有足够规模，或至少可做 low-data 1%/5%/10% 曲线
- 能做 patient-level split（避免泄漏）

**加分（Nice-to-have）**

- 多机构/多设备（更强 domain shift）
- 多任务或多模态（CXR+CT，或可选 MRI）

### 1.0.1 三类核心数据集/监督信号分别是什么意思（重点解释前两个）

> 你可以把这三类数据理解为：**规模（CXR） + 强监督密集任务（CT segmentation） + 可靠性卖点（measurement）**。它们共同支撑你论文的证据链：external/low-data/可靠性。

#### A) CXR 大规模弱监督分类（必选）

它是什么：
- CXR = chest X-ray（胸片），**2D** 影像。
- “弱监督分类” = 给整张片子打标签（不是像素级 mask），典型标签形态是 `present/absent/uncertain`（例如 CheXpert）。
- 标签往往来自报告抽取/规则/弱标注，因此叫 weak supervision。

为什么你需要它（本方法视角）：
- 它承担“医学版 ImageNet”的角色：**大规模 + 多样性**，让 ViT backbone 真正学到稳健表征。
- `uncertain`/不可答在医学里是常态，天然契合你 `uncertainty/answerability/abstain` 的可靠性闭环。

你会怎么用它映射到 UMS-JSON：
- `modality="CXR"`
- `findings`：把每个疾病标签映射成 `present/absent/uncertain`（写进 `findings.<name>.state`）
- `study_view`：如果元数据提供（AP/PA/LAT），就填；否则 `null + answerability=false`
- `laterality`：多数情况下不可可靠推断 → `null + answerability=false`（或只在可确定规则下填写）
- `geometry/measurements`：通常 `null`（可选弱 ROI 只能用于正则，不当真值）

#### B) CT 体素级分割（必选，MICCAI 证据强项）

它是什么：
- CT 是 **3D 体数据**（一组切片组成 volume）。
- “体素级分割” = 每个体素都有 label（器官/肿瘤/结构），数据集提供 3D mask（如 AMOS、KiTS21）。

为什么你需要它（本方法视角）：
- 分割是密集预测任务，对空间结构/边界/形态依赖更强，是证明“表征更基础、更可迁移”的硬证据。
- 它能让你的论文从“只做分类预训练”升级为“覆盖密集任务的医学表征学习框架”，更符合 MICCAI 期待。

你会怎么用它映射到 UMS-JSON（两层用途，建议都写进论文）：

1) **训练监督（结构化 token 目标）**：把 mask 转成可验证的结构化属性（确定性编译，不让 LLM 编）
   - `anatomy`：器官列表（来自数据集标注）
   - `geometry`：可以填 bbox / mask 引用（例如路径/URI；大体积不直接塞 JSON）
   - `measurements`：由 `mask + spacing` 确定性计算（体积/面积/最长径等），并写清 `spacing_source`
   - `laterality`：如果你没做可靠推导规则，默认 `null`（避免硬填带来噪声）

2) **ViT-only 下游迁移评测（主证据）**：训练后丢掉 LLM，把 ViT 接分割 head，在 AMOS→KiTS21 做 transfer + low-data 曲线（Dice/HD95）。

#### C) 数值测量类 CT（强烈建议，加分项）

它是什么：
- 提供“可量化测量”的 CT 数据（如结节直径、肿瘤大小、属性），典型例子是 LIDC-IDRI。

为什么强烈建议：
- 你的卖点之一是 **clinical hard errors**（单位错、量纲×10、范围错）显著下降；没有“数值+单位”的任务，这个卖点很难打到审稿人。

你会怎么用它映射到 UMS-JSON：
- 把测量写进 `measurements.<name> = {value, unit, state, spacing_source}`，并用 verifier 做 `unit/range/×10` 类 hard error 统计。

### 1.1 CXR（2D）

- **训练：CheXpert**
- **外部测试：NIH ChestX-ray14**（明确 domain shift）

用途：

- CheXpert：finding 弱标签（含 uncertain）→ 结构化字段监督；部分 view/元数据 → `study_view`
- NIH：作为 external test（与 CheXpert 标签集合需做对齐子集，具体映射待定）

### 1.2 CT（v1.0：2.5D 三切片为主线）

- **训练：AMOS（CT 部分）**
- **外部迁移：KiTS21**

用途：

- AMOS：有 mask + spacing → measurement 可确定性计算（不是伪标签）；v1.0 建议用 **2.5D（相邻 3 slice 当 3 通道）**，成本小、证据更强
- KiTS21：任务/来源不同，适合作为 transfer 与 low-data 曲线的主证据

### 1.3 可选加分：测量/数值字段

- **LIDC-IDRI（可选）**：结节相关测量/属性回归，非常契合 JSON 数值字段 + unit verifier（是否加入待定）

### 1.4 MRI 需不需要？

- **不强制**：CT + X-ray + external test/transfer 已满足常见 MICCAI 预期。
- **想低成本加分**：可直接使用 **AMOS 自带 MRI 子集**（无需再引入全新 MRI 数据体系；是否加入待定）。

#### 1.6.1 必需数据集（v1.0 最小闭环）

**A) CheXpert（CXR，训练用；采样 30k）**

- 影像：CXR 图像（原始格式按你获取渠道；能对应到 study/patient id）

- 元数据：`study_view`（若该样本缺失 → `null + answerability.study_view=false`）

**B) NIH ChestX-ray14（CXR，external test-only）**

- 影像：CXR 图像
- 标签：NIH findings（用于 external test；不参与训练）
- 映射表（必须自己准备）：`NIH label -> CheXpert label subset` 的对齐表（无法对齐的标签不参与主表）
- 元数据：多数字段可能缺失 → 按 v1.0 口径写 `null + answerability=false`

**C) AMOS（CT，训练用；2.5D 三切片）**

- 影像：CT volume（或官方提供的处理后体数据）
- 标注：多器官 segmentation masks（用于 `geometry`）
- 关键元数据：spacing（DICOM 或 dataset metadata；用于确定性计算 measurement）
- 派生（你需要自己生成）：从 volume 提取 2.5D（三切片）样本索引（slice_id/邻域/spacing），以及由 `mask + spacing` 计算出的 `measurements.*`

**D) KiTS21（CT，transfer 用；含 low-data 子集）**

- 影像：CT volume
- 标注：kidney/tumor masks（用于 `geometry`）
- 关键元数据：spacing（用于 measurement 的确定性计算/校验）
- 划分：官方划分 + 你额外生成的 low-data 划分（1%/5%/10% 的训练子集 ids + seed）
- 派生：与 AMOS 同口径的 2.5D 样本与 `measurements.*`（字段名/口径先保持一致，细节可在 v1.1 再精炼）

#### 1.6.2 可选加分数据集（不阻塞 v1.0）

**E) LIDC-IDRI（CT measurement 加分）**

- 影像：CT volume
- 结节标注/属性/测量：用于 `measurements`（能做 unit/×10/range hard errors）

**F) AMOS MRI 子集（低成本 MRI 加分）**

#### 1.6.3 你必须自己产出的“工程数据”（不然无法训练/评测）

> 这些是你的方法论文里真正“可复现”的部分：UMS 编译、约束解码、verifier 报告。

- **UMS 编译结果（dataset → UMS）**：每条样本一份 UMS-JSON（或打包成 jsonl），字段缺失按 v1.0 口径写 `null + answerability=false`
- **coverage 统计**：更新 `schema/coverage_table.md`（把你实际覆盖的字段与缺失率写清楚）
- **verifier 输出**：`pass/fail + failure_type + confidence + role`（并导出 `verifier_pass_rate.csv`、`failure_taxonomy.csv`）
- **external 标签对齐表**：NIH↔CheXpert 的共同标签子集映射（主表必须用它）

### 1.7 v1.1：统一 ViT 的跨模态混训方案（CXR + CT）

> 这节回答两个问题：  
> 1) “能不能把 CT 和 X 光混在一起训练？”——可以；  
> 2) “怎么混才不互相干扰？”——需要写死采样配方 + 预处理口径 + modality 条件输入。

#### 1.7.1 四个数据集在 v1.1 里分别扮演什么角色？

- **参与预训练（混训）**：`CheXpert（CXR）` + `AMOS（CT）`
- **只用于评测（不参与预训练）**：`NIH（CXR external test-only）` + `KiTS21（CT transfer-only）`

> 这样你才能在论文里把“跨域/迁移”说清楚：external/transfer 必须 test-only。

#### 1.7.2 CT 和 X-ray 差异有多大？会不会“消化不良”？

差异确实大（投影 vs 断层、强度物理意义不同、分辨率/噪声分布不同），所以 v1.1 的关键不是“能不能混”，而是：

- **把 CT 转成 2.5D（3 切片→3 通道）**：让它在 ViT 眼里和 CXR 一样都是 `(B,3,224,224)` 的输入形态
- **统一强度口径**：避免模型靠“强度范围”做域分类捷径
- **用 modality 条件输入**：让模型知道自己在处理哪类影像，减少负迁移

#### 1.7.3 输入格式（统一 2D ViT；CT 用 2.5D，不换结构）

- **CXR**：单张 2D 图 → resize 到 224；归一化（z-score 或 min-max，写死一个口径即可）
- **CT（2.5D）**：
  - 从 volume 取 `slice_{k-1}, slice_k, slice_{k+1}` 拼成 3 通道输入
  - 只监督中间切片 `slice_k` 对应的 mask/字段（与 UMS 目标对齐）
  - CT 强度建议先做 window/clip（如 soft tissue window）再归一化（把口径写进实现与论文）

#### 1.7.4 modality token / prompt（训练与测试要一致，但可自动化）

v1.1 推荐采用 **modality prompt**（最省事、可复现、不会改 ViT 结构）：

- prompt 固定模板里显式包含：`modality = CT` 或 `modality = CXR`
- 推理/评测时同样提供该信息，但**不需要人工输入**：由数据源类型自动选择（DICOM/NIfTI 元数据或数据管线本身就知道）

> 如果你后续想做“learnable modality token”，也可以作为 ablation：在视觉前缀 tokens 前加一个 per-modality learnable token；但 v1.1 主线先用 prompt 版本即可。

#### 1.7.5 两种混训方式（推荐默认：continual + replay；同时混训作为 ablation）

**方案 A（默认推荐）：continual pretraining + replay（更稳）**

1. Stage A（CXR 为主）：用 CheXpert 把生成式结构化监督跑稳（先把 `L_tok` 跑通）
2. Stage B（CT 为主）：加入 AMOS（2.5D）继续训练，同时混入少量 CXR replay 防遗忘  
   - replay 比例建议从 **10%** 起（可在 5–20% 内调）

**方案 B（可做消融/增强）：同时混训（multi-domain mix）**

- 训练从第 1 步就混：按采样概率在 `CheXpert` 与 `AMOS` 之间抽 batch  
  建议起步比例：`CXR:CT = 2:1`（CXR 更容易先收敛），稳定后逐步靠近 `1:1`

#### 1.7.6 你需要在代码里写死的 3 个“配方参数”（否则不可复现）

1. **采样配比**：每阶段 `p(CXR)` / `p(CT)`（以及 replay 比例）
2. **预处理口径**：CXR normalize 方式；CT window/clip + normalize 方式；2.5D 边界 slice 的 padding 策略
3. **modality 条件输入**：用 prompt（v1.1 默认）或 learnable token（ablation）

## 2. UMS 字段覆盖（预期）

> 目的：把“每数据集能提供什么监督”写成**可对齐/可复用的映射表**，为后续实现 `dataset → UMS compiler`、`verifier rules` 与实验对齐口径打底。

表头字段释义（UMS）：

- `modality`：影像模态（如 `CT`、`CXR`、`MRI`）
- `anatomy`：解剖部位/器官（可来自数据集标注或规则推导）
- `findings`：影像所见/病灶标签（如 pneumonia、effusion 等；可含 `present/absent/uncertain`）
- `study_view`：成像体位/投照角度（CXR 常见 `AP/PA/LAT`；没有就设 `null`）
- `laterality`：左右侧（`left/right/bilateral`；不可判定就设 `null`）
- `geometry`：空间标注（`mask/bbox/keypoints` 等，用于定位/分割/测量）
- `measurements`：带单位的数值测量（如直径/面积/体积；最好能由 `geometry + spacing` 确定性计算/校验）

表格符号说明：

- ✅：数据集能提供相对可靠的监督信号（或可确定性推导）
- ⚠️：部分可用/弱信号/需要映射或推导（需在 coverage 表与实现里写清）
- `null`：该字段通常不可答/缺失，按 UMS 规范显式输出 `null`（并配套 `answerability=false`）

字段缺失处理（v1.0 口径）：

- 当数据源缺失 `study_view / laterality / measurements / ...` 等字段时，统一输出 `null`，并设置对应 `answerability.<field>=false`。
- 不引入 `unknown/U` 等新枚举值（保持与 `schema/ums_v0_2.schema.json` 枚举闭包一致），从而让 constrained decoding 与 verifier 口径更可复现。

| 数据集           | modality | anatomy       | findings      | study_view    | laterality    | geometry               | measurements       | 备注                                            |
| ---------------- | -------- | ------------- | ------------- | ------------- | ------------- | ---------------------- | ------------------ | ----------------------------------------------- |
| AMOS             | CT       | ✅            | ⚠️          | `null`      | `null`      | ✅（mask/bbox）        | ✅（mask+spacing） | laterality/view 多数不可答；建议训练 abstain    |
| CheXpert         | CXR      | ⚠️/`null` | ✅            | ✅/⚠️       | ⚠️/`null` | `null`（可选弱 ROI） | `null`           | ROI 仅用于正则/ROI token drop，不当真值         |
| NIH ChestX-ray14 | CXR      | ⚠️/`null` | ✅（需映射）  | `null`/⚠️ | ⚠️/`null` | `null`               | `null`           | external test；与 CheXpert 标签对齐子集（待定） |
| KiTS21           | CT       | ✅            | `null`/⚠️ | `null`      | `null`      | ✅（mask/bbox）        | ✅（mask+spacing） | transfer；measurement 字段名/口径待定           |

### 2.1 AMOS（CT）

- geometry（mask/bbox）：✅
- measurement：✅（由 mask + spacing 确定性计算）
- anatomy：✅（器官类别）
- laterality/view：通常 `null`（多数不可答；应训练 abstain）

### 2.2 CheXpert / NIH（CXR）

- findings：✅（CheXpert 自带；NIH 需映射/对齐）
- study_view：✅/⚠️（取决于元数据）
- geometry/measurement：一般 `null`（可选 CAM/SAM 产生弱 ROI，但**不当真值**，仅用于正则/ROI token drop）
- laterality：谨慎（可推则填，冲突/不可判定则 `null`）

### 2.3 KiTS21（CT）

- geometry（mask/bbox）：✅（分割真值）
- measurement：✅（可由 mask + spacing 计算；字段名待定）
- anatomy：✅（kidney/tumor 等）

## 3. 规模建议（先给可执行默认，细节可在 `11_待确认与留空项.md` 固化）

## 4. 获取渠道与合规写法（要点）

- Kaggle 可以作为下载镜像，但论文必须声明 **origin + license/terms + 获取方式**（不能只写 Kaggle）。
- CheXpert 通常涉及 DUA/协议；细节见本文件末尾“合规、数据声明与伪标签红线”。

## 5. 开工顺序（v1.0 最短闭环）

1. AMOS：CT→2.5D（三切片）+ mask→measurement（确定性脚本）
2. CheXpert：抽样 + 标签映射（含 uncertain）+ view 元数据整理
3. NIH：准备 external test 的标签对齐子集与评测脚本（不必参与训练）
4. KiTS21：准备 transfer 的数据管线与 low-data 子集划分
5. 统一输出到 UMS（JSON）+ `schema/coverage_table.md`
6. verifier v0.2（先 5–8 条规则）+ failure taxonomy 统计

## 6. 合规、数据声明与伪标签红线（v1.0）

### 6.1 数据合规基本原则

- 逐个数据集声明 **origin + license/terms + 获取方式**（MICCAI 常要求 Data Use Declaration）。
- 论文示例图确保不含可识别信息（PHI/叠加文字等）。

### 6.2 伪标签红线（必须写进方法/伦理声明）

#### 6.2.1 红线（对话中明确的原则）

> **凡是“可以从几何/元数据算出来的，绝不让 LLM 编”。**

measurement 必须来自确定性计算（geometry + spacing），而不是 AI 生成。

#### 6.2.2 合法伪标签的条件（让审稿人信）

1. **可解释、可复现**（规则/模型版本固定）
2. **有程序化校验（verifier）**
3. **错误可统计**（failure taxonomy + pass rate）
4. **人工成本可量化且极小**（doctor budget loop，抽检即可）

#### 6.2.3 “AI 永远不直接产 final label”

AI/LLM 只能产候选字段，必须进入：

`schema → verifier → (可选) doctor budget → final JSON`

### 6.3 laterality/view 的敏感性声明（建议写法）

- laterality/view 在部分 2D 场景本质不可判定；我们将其建模为 `answerable=false` 并训练 abstain（输出 null/uncertain）。
- 评测采用 hard error@coverage：在 coverage 受控前提下报告错误上界与拒答收益。

### 6.4 待补信息（占位）

- 各数据集具体许可条款与引用格式：待补
- 数据处理脚本是否发布、发布粒度：待补
